# AES Codec Design — APsystems ECU L1 Encryption

Reverse-engineered from `main.exe`
(ARM:LE:32:v8, ARM Cortex, OpenSSL 1.1.0 dynamic linkage).

This document specifies byte-exact-compatible encrypt + decrypt for the L1
over-the-air ciphertext frames that APsystems main.exe emits and accepts on
fleets running `AES_flag_ALL=1`. It is the spec for a follow-up build agent
to implement in `inv-driver/codec/`.

## TL;DR

There are TWO separate AES schemes in the APsystems firmware. Do not
conflate them:

| Scheme | Endpoints | Key | Status |
|--------|-----------|-----|--------|
| **A. ZB-modem framing** (`readAESzb`/`writeAESzb`) | ECU host CPU ↔ ZigBee modem over `/dev/ttyO4` | STATIC `#6!*Xm&BWt#K3e7J` (16 bytes ASCII) | Already implemented in `ecu-zb/internal/modem` |
| **B. L1 over-the-air** (`AES_deencryption`/`write_AES`) | ECU ↔ inverter, on-air, after the modem has done its job | PER-FRAME derived: `random6 ‖ BCD(invID) ‖ 18 28 45 90` | **NOT IMPLEMENTED. This document specifies it.** |

The two schemes can compose: a single byte stream that leaves the ECU
goes encrypt-A (host→modem) → modem strips A → modem sends OTA → on-air
bytes are encrypt-B → modem at far end strips B → inverter sees
plaintext. Each leg is independent. This document covers **Scheme B
only**.

## Cipher specification (Scheme B)

| Field | Value |
|---|---|
| Algorithm | AES-128 |
| Mode | ECB |
| Key length | 128 bits (16 bytes) |
| Block size | 16 bytes (mandatory; AES-ECB) |
| IV | none (ECB) |
| Padding | **Length-prefixed zero-padding.** The plaintext fed into AES is `[L | body[L] | zeros]` where the first byte `L` is the body length and the remainder is zero-padded so total length is a multiple of 16. There is NO standard PKCS#7 / X.923 padding — the length byte at the start is the framing. |

Ghidra evidence: both `AES_deencryption` @ `0xd944` and `write_AES` @
`0xce80` call `AES_ecb_encrypt(in, out, &key, 0|1)` in a tight per-block
loop, no IV/counter, no chaining. Both call `AES_set_decrypt_key(...,
0x80, ...)` / `AES_set_encrypt_key(..., 0x80, ...)` with `bits=0x80=128`.

## Key derivation (Scheme B)

The AES key is 16 bytes:

```
key[0..6]   = NONCE (6 bytes, random per frame, generated by SENDER)
key[6..12]  = BCD(inverter ID)         (6 bytes — the 12-char ASCII hex ID packed)
key[12..16] = 0x18 0x28 0x45 0x90      (4 fixed bytes — same for every frame, every fleet)
```

The 12-char ASCII inverter ID is packed into 6 bytes via:

```
for i in 0..6:
    bcd[i] = ((id[2i] << 4) + id[2i+1] + 0xD0) & 0xFF
```

`id` is the 12-character ASCII inverter ID (digits `0..9`, possibly `A..F`),
e.g. `"806000042582"`. Verification: `'8'=0x38`, `'0'=0x30`, so
`(0x38<<4)+0x30+0xD0 = 0x480` → low byte `0x80`. Likewise `"60"` → `0x60`.
The packed BCD is therefore identical to the on-wire `PeerUID` already
parsed in `inv-driver/codec/codec.go:ParseL1`.

**The published static key `#6!*Xm&BWt#K3e7J`** (bytes
`23 36 21 2a 58 6d 26 42 57 74 23 4b 33 65 37 4a` at `.rodata
0x000cff3c`) is used ONLY by Scheme A (`readAESzb`/`writeAESzb`). The
"`/dev/ttyO4`" string that the Ghidra decompiler shows concatenated to
it is unrelated neighboring `.rodata` (also at `0xcff4c`); only the first
16 bytes are read because `bits=0x80`. **Scheme B does not use this
key.**

The fixed tail `0x18 0x28 0x45 0x90` is hard-coded in both encrypt and
decrypt routines as four byte stores; it has no symbolic name in the
binary and no XRef.

## Wire layout (Scheme B)

### RX — inverter → ECU (`zigbeeRecvMsg` @ `0x15cc8`, `AES_deencryption` @ `0xd944`)

```
byte offset  field
[0..2)       FC FC                       (L1 SOF — universal RX magic)
[2..4)       short_addr (big-endian)     (16-bit ZigBee short addr)
[4]          RSSI                        (modem-stamped signal byte)
[5]          LQI                         (modem-stamped secondary)
[6..12)      PeerUID                     (6-byte BCD inverter ID = key[6..12])
[12..18)     NONCE                       (6 bytes random, sender-chosen = key[0..6])
[18..N)      CIPHERTEXT                  (multiple of 16 bytes)
```

The RX gate byte is at offset 12. In plaintext frames it is `FB` (the L2
SOF). In encrypted frames it is the first byte of the random nonce, so
its value is uniform-random in `[0x00, 0xFF]`. The dispatcher branches
encrypted iff `frame[12] < 0xF0`. (This means ~6.25% of random nonces
collide with the plaintext-gate band `[0xF0, 0xFF]` and would be
mis-classified as plaintext — the on-wire protocol accepts the false-
negative rate. See "Edge cases" below.)

`AES_deencryption` is called as
`AES_deencryption(frame+6, frame /* out */, total_len - 0x12)`.

After decrypt the routine concatenates all decrypted blocks into a
scratch buffer, then:

```
out[0..N']           = scratch[1 .. 1 + scratch[0]]    (scratch[0] is plaintext length)
if scratch[0] >= 0x5F:
    out[N'..N'+4]    = 00 00 FE FE                     (appends L2 EOF-style tail)
return N' (or N'+4 in the large-plaintext branch)
```

So the **plaintext body** is what was originally a normal L2 frame body
(without the FC-FC L1 envelope, without RSSI, without PeerUID). The
length byte `scratch[0]` is the byte count of that body. The `00 00 FE
FE` tail is a quirk of the large-frame branch and should be reproduced
on emit only for compatibility with consumers that expect it.

### TX — ECU → inverter (`write_AES` @ `0xce80`)

The TX frame is built into a 0x16-byte (22-byte) header followed by
ciphertext:

```
byte offset  field
[0..4)       AA AA AA AA                 (L1 TX SOF — universal TX magic)
[4]          opcode                      (set by caller, e.g. 0xAA/0x1C/0xC3 ...)
[5..7)       short_addr (big-endian)     (set by caller)
[7..11)      00 00 00 00                 (reserved, zero)
[11]         0xA0                        (frame-flag, hard-coded by write_AES)
[12..14)     checksum_be16(frame[4..12]) (signed 16-bit sum)
[14]         total_ciphertext_len + 7    (payload length field)
[15]         GATE: 0xA0 (broadcast: id="000000000000") OR 0xA1 (unicast)
[16..22)     NONCE                       (6 bytes random; copy of key[0..6])
[22..22+C)   CIPHERTEXT                  (C = padded-up multiple of 16)
```

The plaintext fed into AES is:

```
plaintext[0]   = payload_len             (count of bytes of the actual L2 body, not padded)
plaintext[1..] = L2_body                 (payload_len bytes; this is what the caller passes
                                          as `header[0x0F..]` in the un-encrypted code path)
plaintext[N..] = 00 ... 00               (zero-pad to next 16-byte boundary)
```

The `0xA0`/`0xA1` distinction at byte 15: `write_AES` sets `0xA0` iff
the caller's inverter-ID argument equals the string `"000000000000"`
(broadcast). Any other ID → `0xA1` (unicast). Both are `< 0xF0`, so the
receiver's gate check classifies the frame as encrypted in either case.

## Gate-byte semantics

- **RX (inbound) gate:** `frame[12]`. `< 0xF0` ⇒ encrypted; `>= 0xF0`
  (typically `0xFB`, the L2 SOF) ⇒ plaintext.
- **TX (outbound) gate:** `frame[15]`. `0xA0` (broadcast) / `0xA1`
  (unicast) when encrypted. When emitting plaintext the same byte
  position holds the L2 SOF (`0xFB`).

The asymmetry (RX gate offset 12 vs TX gate offset 15) reflects the
extra 3 bytes that the receiving modem strips from the TX envelope (the
modem adds RSSI/LQI/short_addr framing and removes some TX padding —
`zigbeeRecvMsg` only sees what the modem hands up after radio-layer
processing).

Build agents must therefore:
- On RX: gate on `raw[12] < 0xF0`. **Already done** in `codec.ParseL1`
  (`codec/codec.go:79`).
- On TX: explicitly set `raw[15] = 0xA0` (broadcast) or `0xA1`
  (unicast) when emitting an encrypted frame.

## Encrypt pseudocode (Go-ish)

```go
// fixedKeyTail is hard-coded in main.exe write_AES @ 0xce80, lines
// "local_28[local_14 + 6..9] = '\x18'/'('/'E'/0x90".
var fixedKeyTail = [4]byte{0x18, 0x28, 0x45, 0x90}

// EncryptL1 builds an AES-encrypted outbound L1 frame from a plaintext
// L2 body, the destination inverter ID, opcode and short address.
// id is the 12-char ASCII inverter ID; broadcast = "000000000000".
// nonce6 is 6 random bytes (caller-provided so tests can be
// deterministic; in production, crypto/rand.Read).
func EncryptL1(opcode byte, shortAddr uint16, id string, body []byte, nonce6 [6]byte) ([]byte, error) {
    if len(id) != 12 {
        return nil, fmt.Errorf("inverter id must be 12 ASCII chars, got %d", len(id))
    }

    // 1. Build the AES key.
    var key [16]byte
    copy(key[0:6], nonce6[:])
    for i := 0; i < 6; i++ {
        key[6+i] = (id[2*i] << 4) + id[2*i+1] + 0xD0
    }
    copy(key[12:16], fixedKeyTail[:])

    // 2. Build the plaintext: [len|body|zero-pad to 16].
    n := len(body) + 1
    if n%16 != 0 {
        n += 16 - n%16
    }
    pt := make([]byte, n)
    pt[0] = byte(len(body))
    copy(pt[1:], body)
    // remainder is already zero from make()

    // 3. AES-128-ECB encrypt block-by-block.
    cipher, err := aes.NewCipher(key[:])
    if err != nil { return nil, err }
    ct := make([]byte, n)
    for off := 0; off < n; off += 16 {
        cipher.Encrypt(ct[off:off+16], pt[off:off+16])
    }

    // 4. Build the 22-byte L1 header.
    out := make([]byte, 22+n)
    out[0], out[1], out[2], out[3] = 0xAA, 0xAA, 0xAA, 0xAA
    out[4] = opcode
    out[5] = byte(shortAddr >> 8)
    out[6] = byte(shortAddr)
    // out[7..11] stays zero
    out[11] = 0xA0
    // Checksum: signed-arith 16-bit sum of out[4..12], wrap if negative.
    var sum int
    for i := 4; i < 12; i++ {
        sum += int(out[i])
    }
    if sum < 0 { sum += 0xFF }
    out[12] = byte(sum >> 8)
    out[13] = byte(sum % 0x100)
    out[14] = byte(n) + 7
    if id == "000000000000" {
        out[15] = 0xA0   // broadcast
    } else {
        out[15] = 0xA1   // unicast
    }
    copy(out[16:22], nonce6[:])
    copy(out[22:], ct)
    return out, nil
}
```

## Decrypt pseudocode (Go-ish)

```go
// DecryptL1 takes the full inbound frame (FC FC ... ciphertext) and
// returns the plaintext L2 body. Caller has already verified
// raw[12] < 0xF0 (encrypted gate). PeerUID is taken from raw[6..12].
func DecryptL1(raw []byte) ([]byte, error) {
    if len(raw) < 0x13 || raw[0] != 0xFC || raw[1] != 0xFC {
        return nil, fmt.Errorf("malformed L1 frame")
    }
    ctLen := len(raw) - 18
    if ctLen <= 0 || ctLen%16 != 0 {
        return nil, fmt.Errorf("ciphertext length %d not multiple of 16", ctLen)
    }

    // 1. Build the AES key — note the swap vs encrypt:
    //    on the wire we have [PeerUID 6][nonce 6], but the key is
    //    [nonce 6][PeerUID 6][fixed 4].
    var key [16]byte
    copy(key[0:6],   raw[12:18])  // nonce (first 6 bytes of "param_1+6" in main.exe)
    copy(key[6:12],  raw[6:12])   // BCD PeerUID
    copy(key[12:16], fixedKeyTail[:])

    cipher, err := aes.NewCipher(key[:])
    if err != nil { return nil, err }

    // 2. Decrypt block-by-block.
    pt := make([]byte, ctLen)
    for off := 0; off < ctLen; off += 16 {
        cipher.Decrypt(pt[off:off+16], raw[18+off:18+off+16])
    }

    // 3. First byte of plaintext is the payload length; rest is body
    //    (then zero-padding which we drop).
    bodyLen := int(pt[0])
    if 1+bodyLen > len(pt) {
        return nil, fmt.Errorf("decoded length %d exceeds ciphertext %d", bodyLen, len(pt)-1)
    }
    body := make([]byte, bodyLen)
    copy(body, pt[1:1+bodyLen])
    return body, nil
}
```

## Edge cases

1. **Body not a multiple of 16:** handled by the `[len|body|zero-pad]`
   framing. Any body length 0..(ciphertext_blocks*16 - 1) is
   representable.

2. **Gate-byte collision:** Because the gate byte on RX is the first
   byte of the random nonce, ~6.25% of encrypted frames would have a
   nonce starting with a byte in `[0xF0, 0xFF]` and be mis-classified as
   plaintext. Main.exe accepts this; senders are presumably re-rolling
   their nonces if they care, OR the firmware just tolerates the loss
   and retransmits on missing-reply timeout. **Build agents should NOT
   try to reject high-byte nonces on the encrypt side** — that would
   break compatibility with main.exe-sourced frames. On the decrypt
   side, mis-classified frames already fail L2 parse and surface as
   `ErrBadL2SOF`, which is the correct behavior.

3. **Decryption failure / wrong key:** main.exe does NOT downgrade or
   fall back. If decrypt yields garbage, the L2 SOF check
   (plaintext[1..3] should be `FB FB`) will fail at the next layer and
   the frame is dropped. Our codec should error
   (`ErrBadL2SOF`) without retrying.

4. **AES_flag_ALL source:** Per `update_ECU_AES_flag_ALL` @ `0xdbac`,
   `ecu.aes_flag_all` is computed at runtime from the per-inverter
   `aes_flag` (logical AND over all bound inverters). Per-inverter
   `aes_flag` is set in three places:
   - `get_AES` @ `0xcd78`: forced to 1 if the inverter ID's second
     character is `'2'`. (i.e. model-ID prefix indicating a particular
     inverter family — likely DS3 / DSP-series; the user's fleet has IDs
     starting `'80'`/`'70'` so `aes_flag` stays 0.)
   - `set_AES_by_shortaddr` @ `0xce20`: unconditionally sets `aes_flag=1`
     for the matched short_addr (the `aes_value` argument is ignored).
   - `zb_send_cmd` @ `0xfb14`: same `id[1]=='2'` force.

   **No `/etc/yuneng/` config file** controls this; it's purely runtime,
   derived from inverter IDs. The build agent's encrypt-path selector
   should mirror: encrypt iff the target inverter's ID byte 1 is `'2'`,
   OR if `aes_flag` has been explicitly set via a UDS-style API.

5. **Opcode exemptions:** modem-management opcodes (`0x05`/`0x0E`/`0x0F`
   set-PAN, get-info, hardware-reset; see
   `.wolf/memory/zbmodem_startup_sequence.md`) go ECU↔modem only, never
   on-air; they are Scheme A turf, not Scheme B. The current
   `inv-driver` modem startup uses opcode `0x05` and friends through
   Scheme A only. No opcode exemption is needed in the L1 codec for
   Scheme B — every L2 payload bound for an inverter goes through the
   same `EncryptL1` if `aes_flag` is set.

6. **CC2530 vs CC2652 dispatch:** On CC2530-modem ECUs (this fleet),
   `zb_send_cmd` with `aes_flag=1` routes the encrypted frame via
   `write_BRIDGE` @ `0xd78c`, which finds a CC2652-capable peer inverter
   (model `0x32`) to act as a relay. On CC2652-modem ECUs the same path
   calls `write_AES` directly. The build agent does NOT need to model
   the bridge relay — `inv-driver` already speaks L2 directly to
   `ecu-zb` and the modem deals with the radio layer. Just emit the
   `EncryptL1`-shaped envelope and hand it to `ecu-zb`.

## What Ghidra disagreed with vs the user-supplied ground truth

1. **The published key `#6!*Xm&BWt#K3e7J` is NOT the L1 OTA key.** It is
   only the ECU↔modem (Scheme A) static key. The L1 OTA scheme (Scheme
   B — the one that gates on `< 0xF0` and is the user's actual target)
   uses a **per-frame derived key** with a random nonce and the BCD
   inverter ID. The `codec/codec.go` header comment is therefore
   misleading on two counts: (a) it says AES-128-ECB which is correct,
   but (b) it implies the static `#6!*Xm` key — which would not decrypt
   a single L1 frame.

2. **The frame layout in the user's brief is partially wrong.** The
   brief describes `AA AA AA AA <op> <body...> <ck> <tail>` with gate at
   "position N". The actual layout differs by direction:
   - RX (inverter→ECU): `FC FC ... gate=raw[12]`
   - TX (ECU→inverter): `AA AA AA AA ... gate=raw[15]`
   The brief conflates the two and undercounts the header bytes (the TX
   header before the gate byte is 16 bytes including SOF, opcode,
   short_addr, reserved, flag-byte `0xA0`, checksum, length). See "Wire
   layout" above.

3. **Plaintext format includes a 1-byte length prefix.** The brief
   implies "the body is encrypted." More precisely: a single byte
   `len(body)` is prepended, then `body`, then zero-padding to 16. On
   decrypt the length byte is stripped before returning to the L2
   parser.

## What couldn't be determined from static analysis

1. **Real on-wire test vectors.** No fleet on hand has `aes_flag=1` for
   any inverter (all IDs are `'8'`/`'7'`-prefixed). Pre-merge
   verification needs either (a) a `'2'`-prefix inverter, (b) a
   recorded main.exe TX trace from a different fleet, or (c) a
   golden-trace exchange with a willing third party. **Until a real
   capture exists this codec is implementation-correct-by-construction
   but not on-wire-validated.**

2. **The exact semantic of the `00 00 FE FE` tail** appended in the
   large-plaintext (`scratch[0] >= 0x5F`) branch of `AES_deencryption`.
   It looks like an L2-EOF emulation for downstream parsers that expect
   `FE FE`, but no caller of `AES_deencryption` examines those bytes —
   they could be a remnant of an earlier protocol revision. The build
   agent should reproduce them for compatibility but not depend on
   them.

3. **Why `set_AES_by_shortaddr` ignores its `aes_value` argument.** It
   unconditionally sets `aes_flag=1`. This looks like a bug or
   abandoned API; we mirror the firmware by treating the function as
   "set, never clear".

4. **The 0xA0 flag at `frame[11]` on TX.** Always written as `0xA0` by
   `write_AES`, never `0xA1`. Its role is unclear (frame type
   indicator? legacy header?). Cross-check against the plaintext
   `zb_send_cmd` path (`local_218[0xb] = 0`) where the same byte is
   zero — so `0xA0` appears to be a "this is the encrypted variant of
   the L1 envelope" marker, redundant with the gate byte at `frame[15]`.

## Implementation notes for the build agent

- The codec already correctly identifies encrypted frames at
  `codec.L1Envelope.Encrypted` (`codec/codec.go:79`). Wire `DecryptL1`
  into `DecodeReplyFromEnvelope` to replace the `ErrEncrypted` short-
  circuit at `codec/codec.go:255-257`.
- Plumb an `id` lookup from `env.PeerUIDString()` to the encrypt path
  (encrypt needs the ASCII id, RX-decrypt builds the key purely from
  the on-wire bytes and doesn't need a lookup).
- Use `crypto/aes` from the Go standard library — no third-party deps.
- Unit-test fixtures live in `codec/aes_test.go`; include at minimum:
  - One encrypt-then-decrypt round-trip with a fixed nonce.
  - One spec-conformance vector showing the BCD encoding of an ASCII
    inverter ID (`"806000042582"` → `80 60 00 04 25 82`).
  - One mis-keyed decrypt asserting `ErrBadL2SOF` (or equivalent).
- DO NOT introduce a CLI flag for "encrypt mode." Mirror the firmware:
  encrypt when the per-inverter `aes_flag` is set (driven by ID prefix
  `'2'` or an explicit settings entry).
